[.slide,data-transition="zoom"]
== Lambdas : Le Coût Caché

Chaque lambda → *objet anonyme* + appel indirect

[source,kotlin]
----
list.filter { it > 0 }.map { it * 2 }
// Devient : 2 objets Function1 + appels invoke()
----

→ Overhead mesurable dans les boucles chaudes

[NOTE.speaker]
--
Ce qui semble anodin en Kotlin — une simple lambda — crée en réalité
un objet anonyme en bytecode. Dans une boucle à 100 000 itérations,
ça fait 100 000 allocations. Ce n'est pas gratuit.
--

[.slide,data-transition="convex"]
== `inline` — Le Compilateur Substitue

[source,kotlin]
----
inline fun mesure(block: () -> T): T {
    val t = System.nanoTime()
    return block().also { println("durée = ...") }
}
----

Le corps de `mesure` et sa lambda sont *copiés* au point d'appel.
Zéro objet, zéro indirection.

[NOTE.speaker]
--
Avec `inline`, le compilateur prend le code de la fonction inline
et de sa lambda, et le copie directement là où vous l'appelez.
Un peu comme un `#define` en C, mais type-safe.
--

[%auto-animate,auto-animate-easing="ease-in-out",auto-animate-duration=0.8]
== Bonnes Pratiques `inline`

[%step]
* ✅ Courtes fonctions d'ordre supérieur
* ✅ `let`, `apply`, `run`, `with`, `also` (stdlib)
* ❌ Fonctions sans lambda
* ❌ Grosses fonctions → code bloat

[NOTE.speaker]
--
La règle est simple : inline est fait pour les lambdas courtes.
La stdlib Kotlin — let, apply, run, with, also — tout est en inline.
Mais n'en abusez pas, sinon votre bytecode explose en taille.
--

== `noinline` et `crossinline`

[source,kotlin]
----
inline fun exec(
    before: () -> Unit,
    noinline after: () -> Unit  // pas inliné
) {
    before()
    Thread { after() }  // doit rester un objet
}
----

[%step]
* `noinline` → la lambda survit hors du scope
* `crossinline` → interdit `return` non-local

[NOTE.speaker]
--
Si vous passez une lambda à un autre thread, vous ne pouvez pas l'inliner
— elle doit survivre en tant qu'objet. C'est là que `noinline` intervient.
`crossinline` lui, garantit que votre lambda ne fera pas un `return`
qui sortirait de la fonction appelante. Plus rare, mais essentiel.
--

== Effacement de Type JVM

[source,kotlin]
----
fun <T> printType(v: T) {
    println(T::class)  // ❌ "Cannot use 'T' as reified"
}
----

La JVM *efface* les génériques → `T` n'existe plus au runtime

[NOTE.speaker]
--
C'est le fameux "type erasure". Java a introduit les génériques en 2004
mais pour rester rétrocompatible, les types sont effacés après compilation.
Résultat : vous ne pouvez pas faire `T::class` au runtime.
Kotlin hérite de cette limitation... mais a une solution.
--

[.slide,data-transition="concave"]
== `reified` — Garder T au Runtime

[source,kotlin]
----
inline fun <reified T> printType(v: T) {
    println(T::class)          // ✅
    println(v is String)       // ✅
}
----

Le compilateur remplace `T` par le type concret.

[NOTE.speaker]
--
`reified` combiné à `inline` résout le problème. Comme `inline` copie
le code au point d'appel, le compilateur connaît le type concret de `T`
et peut le substituer. Vous écrivez `T::class`, le compilateur écrit
`String::class`. Magique, mais ça ne marche qu'avec `inline`.
--

[%auto-animate,auto-animate-easing="ease-in-out",auto-animate-duration=0.8]
== `reified` — Cas d'Usage

[source,kotlin]
----
inline fun <reified T> List<*>.filterIsInstance() =
    filter { it is T }

val mixed = listOf("a", 1, "b", 2)
mixed.filterIsInstance<String>()
// → ["a", "b"]
----

[NOTE.speaker]
--
L'exemple classique : filtrer une liste hétérogène par type.
Sans `reified`, vous devriez passer `String::class` en paramètre.
Avec `reified`, le code est naturel et lisible.
La stdlib utilise déjà cette technique partout.
--

== Résumé

[%step]
* `inline` → supprime l'overhead objet des lambdas
* `noinline` / `crossinline` → contrôle précis
* `reified` → type générique accessible au runtime

*Mesurez avant d'optimiser !*

[NOTE.speaker]
--
Pour résumer : `inline` élimine le coût des lambdas, `reified` vous
rend le type générique au runtime. Ce sont des outils puissants mais
spécifiques. Mon conseil : mesurez vos perfs avant d'optimiser, et
laissez la stdlib faire le travail pour vous. Merci !
