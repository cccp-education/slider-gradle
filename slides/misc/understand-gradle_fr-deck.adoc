=== Qu'est-ce que Gradle ?

Gradle est un *outil d'automatisation de build* open-source reconnu
pour sa flexibilité et ses performances.

[%step]
* Automatise compilation, test, publication, déploiement
* Principalement utilisé pour les langages JVM
* Supporte un large éventail de plateformes via son écosystème de plugins
* Aide à gérer le cycle de vie complet d'un projet logiciel

[NOTE.speaker]
--
Gradle est un outil d'automatisation de build open-source reconnu
pour sa flexibilité et ses performances. Il automatise le processus
de compilation, de test, de publication et de déploiement de logiciels.
--

[.notes]
Mettre l'accent sur l'approche "code-first" (DSL) et l'utilisation
polyvalente au-delà du simple build.

=== Les Concepts Clés de Gradle

Pour maîtriser Gradle, il est essentiel de comprendre ses concepts
fondamentaux.

*Chaque build est composé de projets, eux-mêmes composés de tâches.*

[NOTE.speaker]
--
Pour maîtriser Gradle, il faut comprendre : projets, tâches, plugins et DSL.
--

[.notes]
Les plugins étendent les fonctionnalités et sont configurés via un DSL
(Groovy ou Kotlin).

[%auto-animate,auto-animate-easing="ease-in-out",auto-animate-duration=1.0]
=== Les Concepts Clés — Articulation

[%step]
* *Projet* : Une entité à construire (application, bibliothèque)
* *Tâche* : Une unité de travail atomique (compiler, tester, copier)
* *Plugin* : Ajoute des tâches et des conventions préconfigurées
* *DSL* : La grammaire spécifique pour décrire et configurer le build

[NOTE.speaker]
--
Chaque concept s'articule avec les autres pour former un système cohérent.
--

[.notes]
Schématiser les relations entre projets, tâches et plugins.

=== Structure d'un Projet Gradle

Un projet Gradle suit une *structure conventionnelle* :

[source,text]
----
mon-projet/
├── build.gradle(.kts)   ← configuration du build
├── settings.gradle(.kts) ← projets multi-modules
├── gradlew / gradlew.bat ← Wrapper (Linux / Windows)
└── src/
    ├── main/java/        ← code source
    └── test/java/        ← tests
----

[NOTE.speaker]
--
Un projet Gradle typique suit une structure conventionnelle qui facilite
la gestion. Les fichiers clés sont build.gradle, settings.gradle, et src/.
--

[.notes]
La convention plutôt que la configuration : Gradle sait où trouver
le code si vous suivez cette structure.

=== Le Gradle Wrapper

Le *Gradle Wrapper* est un script essentiel.

[%step]
* Permet d'exécuter un build avec la version spécifiée de Gradle
* *Sans avoir à installer Gradle manuellement*
* Garantit la cohérence des builds entre environnements
* Indispensable en CI/CD

[source,shell]
----
./gradlew build    # au lieu de "gradle build"
----

[NOTE.speaker]
--
Le Gradle Wrapper garantit que tout le monde utilise la même version
de Gradle, sans installation manuelle.
--

[.notes]
Le Wrapper télécharge automatiquement la bonne version si nécessaire.
Facilite l'intégration pour les nouveaux contributeurs.

=== Gérer les Dépendances

Gradle excelle dans la gestion des dépendances.

[source,kotlin]
.Déclaration dans build.gradle.kts
----
dependencies {
    implementation("org.springframework.boot:spring-boot-starter-web:3.2.0")
    testImplementation("org.junit.jupiter:junit-jupiter-api:5.10.0")
}
----

[%step]
* `implementation` : dépendance de compilation + runtime
* `testImplementation` : dépendance pour les tests uniquement
* Résolution automatique depuis *Maven Central*

[NOTE.speaker]
--
Gradle permet de déclarer facilement les bibliothèques nécessaires,
résolues automatiquement depuis Maven Central.
--

[.notes]
Expliquer les scopes : implementation, testImplementation, runtimeOnly, etc.

=== Exécuter des Tâches Courantes

Gradle simplifie l'exécution des tâches grâce à des commandes claires.

[source,shell]
----
./gradlew tasks    # Liste toutes les tâches disponibles
./gradlew build    # Compile + teste + package
./gradlew clean    # Nettoie le répertoire de build
./gradlew test     # Exécute uniquement les tests
----

[NOTE.speaker]
--
gradlew build compile et teste votre projet, gradlew clean supprime
les fichiers générés.
--

[.notes]
Démonstrer ou lister les commandes fréquentes.

[%auto-animate,auto-animate-easing="ease-in-out",auto-animate-duration=0.8]
=== Commandes Essentielles — Résumé

Avec ces *4 commandes*, vous couvrez 90% du quotidien Gradle :

[%step]
* `./gradlew tasks` — découvrir
* `./gradlew build` — construire
* `./gradlew test` — tester
* `./gradlew clean` — nettoyer

[NOTE.speaker]
--
Avec ces quelques commandes, vous êtes opérationnel au quotidien.
--

[.notes]
Encourager à explorer `./gradlew tasks --all` pour découvrir l'écosystème.

=== Pourquoi choisir Gradle ?

[%step]
* *Flexibilité* — DSL Groovy ou Kotlin, puissant et expressif
* *Performance* — Cache de build, exécution incrémentale, parallélisation
* *Extensibilité* — Large communauté et écosystème de plugins
* *Scalabilité* — Gestion efficace des builds multi-projets

[NOTE.speaker]
--
Flexibilité, performance, extensibilité, scalabilité — Gradle est l'outil
de build moderne par excellence.
--

[.notes]
Détailler chaque bénéfice avec des exemples concrets.
