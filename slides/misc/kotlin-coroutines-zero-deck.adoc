== What are Kotlin Coroutines?

Coroutines are a *lightweight, non-blocking* solution for asynchronous
programming in Kotlin.

[%step]
* Not threads — they are *suspending computations*
* 100,000 coroutines vs 100,000 threads: orders of magnitude lighter
* Write async code that *looks synchronous*
* Built into the language, not a library

[source,kotlin]
----
suspend fun fetchUser(id: String): User {
    val response = httpClient.get("/users/$id")  // suspends, doesn't block
    return parseUser(response)
}
----

[NOTE.speaker]
--
Introduce coroutines as a lightweight, non-blocking solution for
asynchronous programming. Contrast with traditional threading models
and callback-based approaches.
--

Discuss "blocking vs suspending". Emphasize coroutines are not threads
but a framework for managing concurrent execution.
Ref: kotlinlang.org/docs/coroutines-overview.html

== Core Concepts: Suspending Functions

The `suspend` keyword marks a function that can *pause and resume*.

[source,kotlin]
----
suspend fun fetchAndProcess(id: String): Data {
    val user = fetchUser(id)           // suspension point
    val orders = fetchOrders(user.id)  // another suspension point
    return process(user, orders)
}
// Looks synchronous, executes asynchronously!
----

*Behind the scenes:* the compiler transforms it into a *state machine*
with continuation-passing style.

Suspending functions can pause their execution without blocking
the underlying thread.

== Suspending Functions — State Machine

What the compiler generates (simplified):

[source,kotlin]
----
// Your code:
suspend fun fetchAndProcess(id: String): Data {
    val user = fetchUser(id)          // label 0
    val orders = fetchOrders(user.id) // label 1
    return process(user, orders)      // label 2
}

// Compiler generates a state machine with labels 0, 1, 2
// Each suspension point returns COROUTINE_SUSPENDED
// The dispatcher resumes at the next label
----

== Coroutine Builders: `launch` vs `async`

[source,kotlin]
----
// launch — fire and forget, returns Job
val job: Job = scope.launch {
    println("Starting background work...")
    delay(1000)
    println("Done!")
}

// async — computes a result, returns Deferred<T>
val deferred: Deferred<Int> = scope.async {
    delay(1000)
    42  // the result
}
val result = deferred.await()  // 42
----

[%step]
* `launch` → `Job` (no result, can be cancelled)
* `async` → `Deferred<T>` (has result, accessed via `await()`)

[NOTE.speaker]
--
Differentiate the two primary coroutine builders.
launch for fire-and-forget, async for computing results.
--

Discuss typical use cases: background tasks vs parallel computations.

== Coroutine Scope & Context

`CoroutineScope` manages the *lifecycle* of coroutines.

[source,kotlin]
----
// In Android:
viewModelScope.launch {     // auto-cancelled when ViewModel is cleared
    // ...
}

// Custom scope:
val scope = CoroutineScope(Dispatchers.IO + SupervisorJob())
scope.launch {
    // survives as long as scope is active
}
scope.cancel()  // cancels ALL children
----

== Scope & Context — Structured Concurrency

Structured concurrency means: *when a scope is cancelled, everything
inside is cancelled.*

[source,kotlin]
----
scope.launch {
    launch { doWork1() }
    launch { doWork2() }
    async  { computeResult() }
}
scope.cancel()  // ← Cancels ALL 3 coroutines, no leaks
----

[%step]
* No stranded coroutines
* Automatic cancellation propagation
* Clear ownership: parent owns children

[NOTE.speaker]
--
Structured concurrency means: when a scope is cancelled,
all its coroutines are cancelled. No leaks, no zombies.
--

== Dispatchers: Where Coroutines Run

`CoroutineDispatcher` determines the *thread or thread pool*.

[%step]
* `Dispatchers.Main` — UI thread (Android, JavaFX)
* `Dispatchers.IO` — disk/network operations (64 thread pool)
* `Dispatchers.Default` — CPU-bound work (CPU core count pool)
* `Dispatchers.Unconfined` — runs in caller thread until first suspension

[source,kotlin]
----
withContext(Dispatchers.IO) {
    val data = fetchFromNetwork()  // runs on IO thread
}
// Back to previous dispatcher
updateUI(data)  // runs on Main thread
----

[NOTE.speaker]
--
Explain the role of CoroutineDispatcher. Detail common dispatchers.
withContext is the idiomatic way to switch dispatchers.
--

[.notes]
Appropriate use cases: IO for network/disk, Default for CPU,
Main for UI, Unconfined for testing.

== Structured Concurrency: Safety First

[source,kotlin]
----
// coroutineScope — all-or-nothing
suspend fun processAll(items: List<Item>) = coroutineScope {
    items.map { item ->
        async { process(item) }  // if ANY fails, ALL are cancelled
    }.awaitAll()
}

// supervisorScope — isolate failures
suspend fun processBestEffort(items: List<Item>) = supervisorScope {
    items.map { item ->
        async { process(item) }  // failures are isolated
    }.awaitAll()
}
----

[NOTE.speaker]
--
Highlight the benefits of structured concurrency: automatic cancellation
propagation, simplified error handling, preventing coroutine leaks.
--

== Error Handling in Coroutines

Strategies for handling exceptions:

[source,kotlin]
----
// 1. try-catch inside coroutine
scope.launch {
    try {
        riskyOperation()
    } catch (e: Exception) {
        handle(e)
    }
}

// 2. CoroutineExceptionHandler (root coroutines only)
val handler = CoroutineExceptionHandler { _, exception ->
    log.error("Uncaught: $exception")
}
scope.launch(handler) {
    throw RuntimeException("Oops!")  // caught by handler
}
----

[NOTE.speaker]
--
Cover strategies for handling exceptions in coroutines:
try-catch for local handling, CoroutineExceptionHandler for global.
--

== Error Handling — Best Practices

[%step]
* Use *`supervisorScope`* for UI — one failure shouldn't break everything
* Use *`coroutineScope`* for transactions — all-or-nothing semantics
* `launch` + `CoroutineExceptionHandler` for top-level error catching
* `async` exposes exceptions via `await()` — handle with `try-catch`

[source,kotlin]
----
val result: Result<Data> = runCatching {
    async { fetchData() }.await()
}
result.onSuccess { updateUI(it) }
      .onFailure { showError(it) }
----

[NOTE.speaker]
--
Use supervisorScope for UI, coroutineScope for transactions.
Always handle exceptions at the right level.
--

[.notes]
Pattern: use runCatching or Result type within coroutines
for predictable error handling.

== Testing Coroutines with `runTest`

[source,kotlin]
----
// Add dependency:
// testImplementation("org.jetbrains.kotlinx:kotlinx-coroutines-test")

class UserServiceTest {
    @Test
    fun `fetchUser returns user data`() = runTest {
        val service = UserService(mockApi)
        val user = service.fetchUser("123")  // suspend call
        assertEquals("John", user.name)
    }

    @Test
    fun `timeout triggers after delay`() = runTest {
        val job = launch { delay(5000) }
        advanceTimeBy(5000)  // virtual time — instant!
        assertEquals(true, job.isCompleted)
    }
}
----

[NOTE.speaker]
--
Introduce `kotlinx-coroutines-test` and `runTest` as the standard
way to write unit tests for suspending functions.
--

[.notes]
Virtual time control makes tests fast and deterministic.
`advanceTimeBy`, `advanceUntilIdle` — all instant.
